# PPTX Planning Mode — 프레젠테이션 기획 프레임워크

> "슬라이드를 만들기 전에, 슬라이드를 설계하라."

이 문서는 `/lg-pptx` 스킬의 **플래닝 모드**를 정의합니다.
사용자가 "PPT 만들어줘", "보고서 작성해줘" 등을 요청하면
**즉시 코드 생성에 들어가지 말고** 이 프레임워크를 따라 기획 → 검증 → 생성 순서로 진행합니다.

---

## 플래닝 모드 활성화 조건

다음 중 하나라도 해당되면 자동 활성화:

| 조건 | 예시 |
|------|------|
| 슬라이드 수가 5장 이상 예상 | "제안서 만들어줘", "전략 보고서" |
| 주제만 있고 구체적 내용이 없음 | "AI 도입 전략 PPT 만들어줘" |
| 사용자가 명시적으로 기획 요청 | "구성안 먼저 잡아줘", "계획부터" |
| 복잡한 보고서 유형 | 제안서, 경영보고, 전략기획, 프로젝트 계획 |

**비활성화 조건** (바로 생성):
- 슬라이드 1-3장의 단순 요청
- 특정 슬라이드 타입만 요청 ("SWOT 분석 슬라이드 하나 만들어줘")
- 이미 상세한 내용이 제공됨
- 사용자가 "바로 만들어줘"라고 명시

## Phase 1: 디스커버리 인터뷰 (Discovery)

### 목적
슬라이드 기획에 필요한 핵심 정보를 **5가지 강제 질문**으로 추출한다.
모호한 답변에는 한 번 더 밀어붙인다 (Push Once, Push Twice).

### 5가지 강제 질문 (Forcing Questions)

사용자의 요청 유형에 따라 질문을 선택적으로 적용한다:

#### Q1: 목적과 청중 (Purpose & Audience)
> "이 발표를 듣는 사람은 누구이고, 발표 후 어떤 행동을 기대하나요?"

- **Push**: "경영진 보고"라고 하면 → "의사결정권자가 누구인지, 이 보고 후 어떤 결정을 내리길 원하는지 구체적으로 알려주세요"
- **분류**: 
  - 정보 전달형 (교육, 공유) → 논리적 흐름, 상세 내용
  - 의사결정형 (보고, 제안) → 핵심 메시지 먼저, 근거 뒤따름
  - 설득형 (제안서, 영업) → 문제-해결-효과 구조

#### Q2: 핵심 메시지 (Core Message)
> "이 PPT를 한 문장으로 요약하면 뭔가요? 청중이 딱 하나만 기억한다면?"

- **Push**: "AI 도입이 필요하다"라고 하면 → "왜 지금이고, 왜 이 방식이고, 안 하면 어떤 리스크가 있나요?"
- 핵심 메시지가 명확하지 않으면 슬라이드가 산만해짐

#### Q3: 콘텐츠 소스 (Content Sources)
> "이미 있는 자료가 있나요? (기존 문서, 데이터, 이미지, 참고 PPT)"

- **Push**: "없다"라고 하면 → "그러면 제가 내용을 구성해도 될까요? 특별히 반드시 포함해야 하는 수치나 사실이 있나요?"
- 첨부 파일이 있으면 분석하여 콘텐츠 추출
- 웹 리서치가 필요하면 `/browse` 활용 가능 여부 확인

#### Q4: 구조 제약 (Constraints)
> "분량, 발표 시간, 반드시 포함할 섹션이 있나요?"

- **Push**: "적당히"라고 하면 → "10분 발표면 10-12장, 30분이면 20-25장, 읽기용 보고서면 30장+ 가 적정합니다. 어느 쪽인가요?"
- 목표 분량 → 슬라이드 배분 전략 결정

#### Q5: 스타일 선호 (Style Preference)
> "시각적 임팩트 위주인가요, 데이터 중심인가요, 균형형인가요?"

- **Push**: "깔끔하게"라고 하면 → "구체적으로: (A) 인포그래픽처럼 비주얼 강조 (B) 컨설팅 보고서처럼 내용 빽빽 (C) 적당한 균형 — 어떤 느낌인가요?"
- 스타일 → density 프리셋 매핑:
  - 비주얼 임팩트 → `spacious`
  - 균형형 → `normal`
  - 내용 중심 → `compact`
  - 제안서/보고서 → `dense`

### 스마트 라우팅 (질문 생략 규칙)

| 이미 알고 있는 정보 | 생략 가능한 질문 |
|---------------------|------------------|
| 첨부 파일로 내용 제공됨 | Q3 |
| "15장짜리 제안서" 등 분량 명시 | Q4 |
| "컨설팅 스타일로" 등 스타일 명시 | Q5 |
| 전부 명확함 | 전부 생략 → Phase 2로 직행 |

### 인터뷰 규칙

1. **한 번에 하나씩**: 질문은 한 개씩, AskUserQuestion 도구 사용
2. **금지어**: "interesting", "좋은 생각이네요", "여러 가지 방법이 있는데" — 구체적 제안만
3. **입장 표명**: 항상 추천안을 제시한 뒤 선택지를 줌
4. **5분 룰**: 인터뷰는 5개 질문 이내로 종결, 질질 끌지 않음

## Phase 2: 슬라이드 아키텍처 (Slide Architecture)

### 목적
인터뷰 결과를 바탕으로 **슬라이드별 구조표**를 작성한다.
각 슬라이드가 어떤 빌더 메서드를 사용하고, 어떤 내용을 담는지 명시한다.

### 슬라이드 구조표 형식

```markdown
| # | 제목 | 타입 | 메서드 | Density | 핵심 내용 |
|---|------|------|--------|---------|-----------|
| 1 | 프로젝트명 | 표지 | add_cover | - | 제목, 팀명, 날짜 |
| 2 | Contents | 목차 | add_toc | - | 4개 섹션 |
| 3 | I. 배경 | 섹션구분 | add_section_divider | - | 섹션 1 |
| 4 | 1.1 현황 분석 | 내용 | add_content | normal | 현황 불릿 5개 |
| 5 | 1.2 문제점 | 2단비교 | add_two_column | normal | As-Is vs To-Be |
| 6 | II. 전략 | 섹션구분 | add_section_divider | - | 섹션 2 |
| 7 | 2.1 추진 전략 | 타이틀컬럼 | add_titled_columns | normal | 3개 전략 축 |
| 8 | 2.2 로드맵 | 로드맵 | add_roadmap | compact | 3개년 로드맵 |
| ... | ... | ... | ... | ... | ... |
```

### 아키텍처 설계 원칙

#### 1. 내러티브 흐름 (Narrative Arc)
프레젠테이션은 이야기다. 다음 구조 중 하나를 따른다:

| 패턴 | 구조 | 적합한 유형 |
|------|------|-------------|
| **문제-해결** | 배경→문제→솔루션→효과→제언 | 제안서, IT 도입 |
| **현황-전략** | 현황→분석→전략→실행계획→기대효과 | 전략 보고서 |
| **보고형** | 요약→실적→이슈→계획→부록 | 주간/월간 보고 |
| **교육형** | 개요→핵심개념→상세→실습→정리 | 교육/세미나 |
| **비교형** | 배경→옵션분석→비교→추천→다음단계 | 솔루션 비교 |

#### 2. 레이아웃 다양성 체크 (Variety Check)
**절대 금지**: 같은 레이아웃이 3장 연속 반복

| 체크 항목 | 기준 |
|-----------|------|
| 연속 동일 레이아웃 | 최대 2장 (3장부터 경고) |
| 전체 중 불릿 슬라이드 비율 | 40% 이하 |
| 시각적 슬라이드 (차트/다이어그램/카드) | 최소 30% |
| 테이블 연속 | 최대 2장 |

#### 3. 밀도 일관성 (Density Consistency)
- 같은 섹션 내에서는 동일 density 유지
- 섹션 간 밀도 변화는 OK (예: 전략 섹션=spacious, 실행 섹션=compact)

#### 4. 슬라이드 배분 가이드

| 전체 분량 | 표지/목차/섹션 | 내용 슬라이드 | 제언/클로징 |
|-----------|---------------|--------------|------------|
| 10장 | 3장 (30%) | 6장 (60%) | 1장 (10%) |
| 15장 | 4장 (27%) | 10장 (67%) | 1장 (6%) |
| 20장 | 5장 (25%) | 13장 (65%) | 2장 (10%) |
| 30장 | 7장 (23%) | 21장 (70%) | 2장 (7%) |

### 메서드 매핑 가이드

| 표현하고 싶은 것 | 추천 메서드 | density |
|------------------|-------------|---------|
| 핵심 수치/KPI | `add_kpi_cards` | spacious |
| 장단점/비교 | `add_two_column` | normal |
| 3~4개 전략 축/필러 | `add_titled_columns`, `add_strategy_pillars` | normal |
| 상세 데이터 테이블 | `add_table` | compact/dense |
| 타임라인/일정 | `add_roadmap`, `add_gantt_chart`, `add_timeline` | normal |
| 프로세스/워크플로우 | `add_process_flow`, `add_icon_process`, `add_swimlane` | normal |
| 조직/계층 구조 | `add_org_chart`, `add_pyramid` | normal |
| SWOT/리스크 분석 | `add_swot`, `add_risk_matrix` | normal |
| 시장 포지셔닝 | `add_positioning_map` | spacious |
| 핵심 키워드/비전 | `add_keyword_highlight` | spacious |
| 메인+보조 패널 구조 | `add_content_sidebar` | normal/compact |
| 카드형 비교/소개 | `add_card_grid`, `add_comparison_cards` | normal |
| N개 항목 나열 | `add_multi_column` | normal |
| 재무/투자 요약 | `add_financial_summary` | compact |
| 마일스톤 추적 | `add_milestone_tracker` | normal |
| 이미지/스크린샷 | `add_image_slide` | - |
| 데이터 차트 | `add_matplotlib_chart`, `add_chart_slide` | - |
| 권고/제언 | `add_recommendation` | normal |

## Phase 3: 콘텐츠 개발 (Content Development)

### 목적
구조표의 각 슬라이드에 **실제 텍스트 콘텐츠**를 채운다.
"제목만 있고 내용이 빈" 슬라이드가 없어야 한다.

### 콘텐츠 품질 기준

#### 불릿 포인트 작성 규칙
| 규칙 | 나쁜 예 | 좋은 예 |
|------|---------|---------|
| 구체적 수치 포함 | "성과 개선" | "불량률 23% → 8% 감소 (15%p↓)" |
| 행동 동사로 시작 | "AI 시스템" | "AI 기반 실시간 예측 시스템 구축" |
| 한 줄에 한 가지 | "분석 및 설계와 구현" | 각각 별도 불릿 |
| 깊이 있는 내용 | "효율화 추진" | "RPA 도입으로 월 160시간 수작업 자동화" |

#### 콘텐츠 소스별 처리

| 소스 | 처리 방법 |
|------|-----------|
| 사용자 첨부 파일 | 분석하여 핵심 내용 추출, 슬라이드별 배분 |
| 사용자 구두 설명 | 구조화하여 불릿/테이블로 변환 |
| 웹 리서치 필요 | 가용 도구로 최신 데이터 수집 (날짜 명시) |
| 내용 없음 (데모) | 주제에 맞는 현실적 샘플 콘텐츠 생성 |

#### 슬라이드별 콘텐츠 밀도 가이드

| 슬라이드 타입 | 최소 내용량 | 최대 내용량 |
|--------------|------------|------------|
| 표지 | 제목 + 팀명 + 날짜 | + 부제 |
| 내용 (spacious) | 불릿 3개 | 불릿 5개 |
| 내용 (normal) | 불릿 4개 | 불릿 8개 |
| 내용 (compact) | 불릿 6개 | 불릿 12개 |
| 내용 (dense) | 불릿 8개 | 불릿 18개 |
| 테이블 | 3행 | 밀도에 따라 15행+ |
| KPI | 3개 카드 | 5개 카드 |
| 비교 | 좌우 각 3개 불릿 | 좌우 각 8개 불릿 |

## Phase 4: 디자인 리뷰 (Design Review)

### 목적
생성 전에 **7가지 패스**로 구조표를 검증한다.
gstack의 디자인 리뷰 프레임워크를 PPT 맥락에 맞게 적용.

### 7-Pass 디자인 리뷰

#### Pass 1: 정보 아키텍처 (Information Architecture)
- [ ] 전체 내러티브 흐름이 논리적인가?
- [ ] 섹션 구분이 명확한가?
- [ ] 각 슬라이드가 하나의 메시지에 집중하는가?
- [ ] 목차와 실제 구성이 일치하는가?

#### Pass 2: 시각적 다양성 (Visual Variety)
- [ ] 같은 레이아웃 3장 연속 없는가?
- [ ] 불릿 슬라이드가 40% 이하인가?
- [ ] 시각적 슬라이드(차트/다이어그램/카드)가 30% 이상인가?
- [ ] density가 섹션 내에서 일관적인가?

#### Pass 3: 콘텐츠 충실도 (Content Substance)
- [ ] "빈 슬라이드" (제목만 있고 내용 없는)가 없는가?
- [ ] 각 불릿이 구체적이고 행동 지향적인가?
- [ ] 숫자/수치가 포함 가능한 곳에 포함되었는가?
- [ ] 중복 내용이 여러 슬라이드에 반복되지 않는가?

#### Pass 4: 청중 적합성 (Audience Fit)
- [ ] 전문 용어 수준이 청중에게 적합한가?
- [ ] 의사결정자용이면 핵심 메시지가 앞에 오는가?
- [ ] 실무자용이면 상세 내용이 충분한가?
- [ ] "so what?" 테스트 — 각 슬라이드가 청중에게 의미 있는가?

#### Pass 5: 메서드 적합성 (Method Fitness)
- [ ] 각 슬라이드의 내용에 최적의 메서드가 선택되었는가?
- [ ] 비교 데이터에 `add_two_column`/`add_comparison_cards` 사용?
- [ ] 수치 데이터에 `add_kpi_cards`/`add_table` 사용?
- [ ] 프로세스에 `add_process_flow`/`add_swimlane` 사용?

#### Pass 6: 흐름과 전환 (Flow & Transitions)
- [ ] 섹션 구분 슬라이드가 적절히 배치되었는가?
- [ ] 챕터명(chapter)이 일관적으로 설정되었는가?
- [ ] 제언/클로징이 마지막에 있는가?
- [ ] 슬라이드 간 논리적 연결이 자연스러운가?

#### Pass 7: 실행 가능성 (Feasibility)
- [ ] 모든 메서드가 현재 빌더에서 지원되는가?
- [ ] 필수 파라미터가 모두 준비되었는가?
- [ ] 이미지/차트 데이터가 필요한 경우 준비 가능한가?
- [ ] 예상 분량이 요구사항과 일치하는가?

### 리뷰 결과 처리

| 결과 | 액션 |
|------|------|
| 7/7 Pass | Phase 5 진행 |
| 1-2개 Warning | 수정 사항 제시 후 사용자 확인 → Phase 5 |
| 3개+ Warning | 구조표 수정 → Phase 4 재실행 |
| Critical Issue | Phase 2로 회귀 |

## Phase 5: 생성 (Generation)

### 실행 순서

1. **구조표 최종 확인**: 사용자에게 슬라이드 구조표 제시
2. **확인 후 생성**: 사용자 승인 시 코드 작성 및 실행
3. **생성 후 QA**: markitdown으로 텍스트 추출하여 내용 검증
4. **결과 보고**: 생성된 파일 경로 + 슬라이드 수 + 요약

### 코드 생성 규칙

```python
# 표준 생성 패턴
import sys
sys.path.insert(0, '<skill-scripts-path>')
from lg_pptx_builder import LGPresentation

prs = LGPresentation()

# 1. 표지
prs.add_cover("제목", subtitle="팀명", date="2026.04.13")

# 2. 목차
prs.add_toc([("섹션1", ["항목1", "항목2"]), ...])

# 3~N. 내용 슬라이드 (구조표 순서대로)
# ... 각 슬라이드 코드 ...

# N+1. 클로징
prs.add_closing("감사합니다")

prs.save("output.pptx")
```

## 약식 모드 (Quick Mode)

5장 이하의 단순 요청에는 약식 플래닝 적용:

1. Phase 1 생략 (질문 없이 맥락에서 추론)
2. Phase 2-3 통합 (구조+콘텐츠 한 번에)
3. Phase 4 축소 (Pass 2, 3, 5만 체크)
4. Phase 5 즉시 실행

## 사용자 대화 흐름 예시

### 예시 1: 제안서 (전체 플래닝)

```
사용자: "AI 기반 품질관리 시스템 제안서 PPT 만들어줘"

[Phase 1 - 디스커버리]
AI: "이 제안서를 보는 의사결정자가 누구이고, 어떤 결정을 기대하시나요?
     예: 공장장 대상 도입 승인 / IT 임원 대상 예산 확보 / 현업 대상 협조 요청"
사용자: "공장장한테 보여주는 거야. 도입 승인 받으려고."

AI: "핵심 메시지를 한 문장으로 정리하면요?
     제가 보기에는 'AI 가상계측으로 불량률 X% 감소, 연간 Y억 절감 가능'
     같은 방향이 될 것 같은데, 맞나요?"
사용자: "맞아. 불량률 30% 감소 목표야."

AI: "분량은 15장 내외, 컨설팅 보고서 스타일(compact)로 가겠습니다.
     구성안을 보여드릴게요."

[Phase 2 - 슬라이드 아키텍처]
AI: 구조표 제시 (15장)

[Phase 3-4 - 콘텐츠 + 리뷰]
AI: "7가지 디자인 리뷰 통과했습니다. 생성할까요?"
사용자: "응"

[Phase 5 - 생성]
AI: PPT 생성 + 결과 보고
```
